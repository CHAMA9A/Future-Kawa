Je veux améliorer le design du frontend FutureKawa avant de passer à l’étape suivante.

Important :
Le frontend fonctionne déjà.
Ne pas casser la logique existante.
Ne pas modifier les routes API.
Ne pas modifier les backends.
Ne pas ajouter Supabase.
Ne pas ajouter d’authentification.
Ne pas créer de nouveau backend.

Objectif :
Améliorer fortement l’interface pour qu’elle ressemble à un vrai dashboard professionnel de supervision IoT/logistique pour FutureKawa.

Problème actuel :
L’interface affiche uniquement des cartes et des tableaux. Elle est fonctionnelle, mais visuellement trop simple et pas assez professionnelle.

Améliorations attendues :

1. Layout général

* Ajouter une vraie sidebar à gauche.
* Sidebar avec :

  * Logo / nom FutureKawa
  * Dashboard
  * Lots
  * Mesures IoT
  * Alertes
  * Pays
* Garder une seule page si nécessaire, mais avec navigation visuelle par sections.
* Ajouter un header moderne avec :

  * titre de page
  * sous-titre
  * sélecteur de pays
  * statut global du système

2. Hero section / bloc d’introduction
   Ajouter en haut du dashboard un bloc visuel professionnel :

* “Supervision des stocks de café vert”
* phrase courte : “Monitoring centralisé des entrepôts, capteurs IoT et alertes qualité.”
* badge “Backend central connecté”
* badge “MQTT actif”
* badge “Multi-country architecture”

3. Cartes statistiques
   Améliorer les cartes existantes :

* Ajouter des icônes simples
* Ajouter des couleurs cohérentes
* Ajouter une indication courte sous chaque valeur
  Exemples :
* Lots en stock
* Lots FIFO
* Mesures IoT
* Alertes actives
* Pays sélectionné
* Statut service

4. Corriger le statut service
   Actuellement, Colombia affiche “hors ligne” alors que les données sont chargées.
   Le statut service ne doit pas être “hors ligne” si les données API sont récupérées avec succès.
   Règle simple :

* Si les appels API réussissent : afficher “En ligne”
* Si erreur API : afficher “Hors ligne”

5. Ajouter des graphiques simples
   Ajouter au moins 2 graphiques visuels :

* Graphique température / humidité des dernières mesures
* Répartition des alertes par type : TEMPERATURE, HUMIDITY, EXPIRED_LOT

Utiliser une librairie simple comme Recharts si possible.

6. Section résumé pays
   Ajouter une carte “Country overview” avec :

* pays sélectionné
* entrepôt principal
* seuils acceptables du pays
* nombre d’alertes
* dernière mesure reçue

Seuils :
Brazil :

* température acceptable : 26°C - 32°C
* humidité acceptable : 53% - 57%

Ecuador :

* température acceptable : 28°C - 34°C
* humidité acceptable : 58% - 62%

Colombia :

* température acceptable : 23°C - 29°C
* humidité acceptable : 78% - 82%

7. Tables
   Garder les tableaux, mais les rendre plus propres :

* Ajouter une meilleure séparation visuelle
* Ajouter des badges colorés pour les statuts
* Limiter la hauteur des tables avec scroll si beaucoup de données
* Ajouter un message vide propre si aucune donnée :
  “Aucune donnée disponible pour ce pays.”

8. Alertes
   Améliorer la section alertes :

* Créer des cartes ou lignes plus visuelles
* TEMPERATURE en rouge/orange
* HUMIDITY en bleu
* EXPIRED_LOT en violet ou rouge foncé
* Afficher les alertes critiques en haut

9. Design system
   Utiliser un thème plus professionnel :

* fond clair beige/café
* brun foncé pour les titres
* vert pour les statuts OK
* orange/rouge pour les alertes
* bleu pour l’humidité
* cartes blanches avec ombres légères
* coins arrondis
* spacing plus moderne

10. Responsive
    L’interface doit rester lisible sur :

* desktop
* tablette
* petit écran

11. Code
    Garder la structure actuelle :
    frontend/src/components
    frontend/src/pages
    frontend/src/api
    frontend/src/types

Créer de nouveaux composants si nécessaire :

* Sidebar.tsx
* Header.tsx
* HeroSection.tsx
* CountryOverview.tsx
* MeasurementsChart.tsx
* AlertsChart.tsx

12. Ne pas faire

* Ne pas changer les endpoints
* Ne pas supprimer les composants existants si possible
* Ne pas créer de mock data comme comportement principal
* Ne pas modifier les backends pays
* Ne pas modifier Docker sauf si nécessaire pour ajouter une dépendance frontend comme Recharts

Après modification, vérifier :

* npm run build fonctionne
* docker compose build frontend fonctionne
* http://localhost:5173 affiche le nouveau dashboard
* Les données réelles du backend central s’affichent toujours
