On passe à l’étape 9 du projet FutureKawa : Jenkins CI/CD.

Contexte :
Le projet est un monorepo avec :

* backend-country/brazil-service
* backend-country/ecuador-service
* backend-country/colombia-service
* backend-central
* frontend
* docker-compose.yml

Les tests sont déjà en place :

* Brazil service : 19 tests
* Ecuador service : 17 tests
* Colombia service : 14 tests
* Backend central : 23 tests
* Frontend : 22 tests
  Total : 95 tests passing.

Objectif :
Ajouter une configuration Jenkins CI propre pour le projet FutureKawa.

Important :

* Ne pas modifier la logique métier.
* Ne pas modifier les tests existants sauf si strictement nécessaire.
* Ne pas ajouter de déploiement réel maintenant.
* Ne pas ajouter Kubernetes maintenant.
* Ne pas intégrer le capteur réel maintenant.
* Se concentrer uniquement sur Jenkins CI.

Travail attendu :

1. Créer un fichier Jenkinsfile à la racine du projet.

Le pipeline doit contenir des stages clairs :

* Checkout
* Environment information
* Install dependencies
* Generate Prisma clients
* Run backend tests
* Run frontend tests
* Build frontend
* Validate Docker Compose
* Optional Docker build check

2. Le pipeline doit lancer les tests pour chaque service :

backend-country/brazil-service :

* npm ci ou npm install
* npx prisma generate
* npm test

backend-country/ecuador-service :

* npm ci ou npm install
* npx prisma generate
* npm test

backend-country/colombia-service :

* npm ci ou npm install
* npx prisma generate
* npm test

backend-central :

* npm ci ou npm install
* npm test

frontend :

* npm ci ou npm install
* npm run test -- --run
* npm run build

3. Utiliser Jest en mode séquentiel

Les scripts backend utilisent déjà :
jest --runInBand

Garder ce comportement pour éviter les problèmes mémoire sur Windows/Jenkins.

4. Ajouter une validation Docker Compose

Ajouter un stage qui lance :

docker compose config

pour vérifier que le fichier docker-compose.yml est valide.

Si possible, ajouter aussi :

docker compose build

mais rendre cette étape optionnelle ou clairement séparée, car elle peut être lourde selon la machine Jenkins.

5. Créer une documentation Jenkins

Créer :

docs/jenkins-ci.md

Cette documentation doit expliquer :

* objectif du pipeline
* prérequis Jenkins
* plugins utiles
* configuration d’un job Pipeline
* lien avec GitHub
* étapes du pipeline
* commandes exécutées
* comment interpréter un build réussi
* comment interpréter un build échoué

6. Mettre à jour README.md

Ajouter une section :

## CI/CD with Jenkins

Inclure :

* présence du Jenkinsfile
* résumé des stages
* commande principale des tests
* lien vers docs/jenkins-ci.md

cd C:\Users\sav\Downloads\Future-Kawa; foreach ($p in @(".\backend-country\brazil-service", ".\backend-country\ecuador-service", ".\backend-country\colombia-service", ".\backend-central")) { npm --prefix $p test; if ($LASTEXITCODE -ne 0) { exit $LASTEXITCODE } }; npm --prefix .\frontend run test -- --run

8. Sécurité

Vérifier que Jenkinsfile ne contient aucun secret :

* pas de mot de passe
* pas de token
* pas de fichier .env
* pas de clé API

9. Résultat attendu

À la fin, le projet doit contenir :

* Jenkinsfile
* docs/jenkins-ci.md
* README.md mis à jour

Le pipeline doit prouver que :

* les 95 tests passent
* le frontend build correctement
* docker-compose.yml est valide

Ne pas faire de déploiement automatique maintenant.
Ne pas ajouter de Jenkins container obligatoire dans docker-compose.yml sauf si tu estimes que c’est utile pour la documentation, mais le plus important est le Jenkinsfile.
